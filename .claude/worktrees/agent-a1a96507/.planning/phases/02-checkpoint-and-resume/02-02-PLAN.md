    2. Create a module-level mutable state object that the signal handler can access. Place this after the imports, before `parseArgs()`:
       ```typescript
       /** Shared state for signal handler checkpoint saves. Updated by fetchAllReceipts progress. */
       let activeCheckpoint: { fetchedBarcodes: string[]; details: ReceiptDetail[]; summaries: ReceiptSummary[]; progress: FetchProgress } | null = null;
       ```

    3. Create signal handler function:
       ```typescript
       function handleShutdown(signal: string): void {
         console.log(`\n${signal} received. Saving progress...`);
         if (activeCheckpoint) {
           saveCheckpoint({
             ...activeCheckpoint,
             lastUpdated: '',
           });
           console.log(`Progress saved to ${CHECKPOINT_FILE} (${activeCheckpoint.progress.fetched} fetched, ${activeCheckpoint.progress.failed} failed).`);
           console.log('Re-run to resume from checkpoint.');
         }
         process.exit(0);
       }
       ```

    4. Register signal handlers at the beginning of `main()`, before any other logic:
       ```typescript
       process.on('SIGINT', () => handleShutdown('SIGINT'));
       process.on('SIGTERM', () => handleShutdown('SIGTERM'));
       ```

    5. In the fetch branch of main() (the `else` block at line 53 that calls `fetchAllReceipts()`), expose fetch state for the signal handler. Modify the `fetchAllReceipts()` integration:

       Before calling `fetchAllReceipts()`, load any existing checkpoint so the signal handler has state even if interrupted during list fetching:
       ```typescript
       const existingCheckpoint = loadCheckpoint();
       if (existingCheckpoint) {
         activeCheckpoint = {
           fetchedBarcodes: existingCheckpoint.fetchedBarcodes,
           details: existingCheckpoint.details,
           summaries: existingCheckpoint.summaries,
           progress: existingCheckpoint.progress,
         };
         console.log(`Resuming from checkpoint: ${existingCheckpoint.progress.fetched} already fetched.`);
       }
       ```

       After `fetchAllReceipts()` returns, clear activeCheckpoint (fetch is complete, no need for signal handler to save):
       ```typescript
       activeCheckpoint = null;
       ```

    6. Note: The signal handler captures whatever `activeCheckpoint` is set to at interrupt time. During the fetch, `costco.ts` calls `saveCheckpoint()` after each receipt, so the on-disk checkpoint is always current. The signal handler is a safety net for the case where costco.ts is mid-request (between "start fetch" and "save checkpoint").

       To keep `activeCheckpoint` in sync during the fetch, we need costco.ts to update it. BUT modifying costco.ts to accept and update a mutable reference adds coupling. Instead, the signal handler should just load the latest checkpoint from disk (which is saved after each receipt by costco.ts):

       Revise `handleShutdown` to use disk state:
       ```typescript
       function handleShutdown(signal: string): void {
         console.log(`\n${signal} received. Saving progress...`);
         const checkpoint = loadCheckpoint();
         if (checkpoint) {
           console.log(`Progress saved in ${CHECKPOINT_FILE} (${checkpoint.progress.fetched} fetched, ${checkpoint.progress.failed} failed).`);
           console.log('Re-run to resume from checkpoint.');
         } else {
           console.log('No progress to save (no receipts fetched yet).');
         }
         process.exit(0);
       }
       ```

       With this approach, remove the `activeCheckpoint` variable entirely. The signal handler reads the already-saved checkpoint from disk. This is safe because costco.ts saves synchronously after each receipt.

    7. Update the `result` variable usage. Since `fetchAllReceipts()` now returns `{ summaries, details, progress }`:
       ```typescript
       const result = await fetchAllReceipts();
       details = result.details;
       ```
